# Git ワークフロースキル

## 事前確認 (Pre-flight Check)
スキル実行前に必ず `git status` を確認し、未コミットの変更があるか把握すること。

## コミットメッセージのルール (厳守)

コミットメッセージは以下のフォーマットで作成すること。

```text
<1行要約: 変更内容を簡潔に（日本語OK）>

<詳細: 小学生でもわかる言葉で、変更の理由や具体的な内容（日本語OK）>
```

### 詳細メッセージの書き方（必須）
- むずかしい専門用語をできるだけ使わない。
- 1文を短くする（目安: 40文字以内）。
- 「何を直したか」「何がよくなるか」をはっきり書く。
- 可能なら2〜4行で書く。
- NG例: 「リファクタリング」「依存解決」「最適化」だけで終える。
- OK例: 「重くならないように、同じ計算を何回もしない形に直しました。」

**例（推奨）:**
```text
機能Aのバグ修正

空っぽの入力でアプリが止まる問題を直しました。
先に入力チェックを入れたので、エラーが出にくくなります。
```

## 必須フロー (ワークフロー)

### 1. 変更のステージング (`git add`)
- ユーザーへの確認は**不要**。
- `git add .` 等を実行してステージングする。

### 2. コミット前の説明と承認 (`git commit`)
- **コミットを実行する前に**、以下の手順を踏むこと。
  1. `git diff --staged` 等で変更内容を確認する。
  2. ユーザーに対して、今回の変更内容を**「小学生にもわかるように」**噛み砕いて説明する。専門用語を避け、何が変わって何が良くなるのかを伝える。
  3. 作成したコミットメッセージ案（要約＋詳細）を提示する。
  4. 「この内容でコミットして良いですか？」と承認を求める。
- 承認が得られた場合のみ、コミットを実行する。

```bash
git commit -m "<1行要約>

<詳細>"
```

### 3. リモートへの反映 (`git push`)
- **Gitオーナー（@Shoma-DS）の場合**:
  - `git push`（または `git push -u ...`）を実行して良いか確認し、承認を得てから実行する。
- **それ以外のメンバーの場合**:
  - 直接プッシュは禁止。**プルリクエスト (Pull Request)** を作成するように案内する。
  - 「オーナーではないため、プルリクエストを作成してください」と伝える。

### 4. マージ・リベース (`git merge`, `git rebase`)
- 実行前に必ずユーザーに確認し、承認を得ること。
- 「なぜマージ/リベースが必要か」を説明する。

## エラー時の対応
- 競合（コンフリクト）が発生した場合は、ユーザーに手動解決を依頼する。


## 注意事項
- コミットメッセージは日本語でも英語でもOK
- プッシュ先は原則として `origin` (team-info) を使用する
- プッシュ時に認証が必要な場合がある（ユーザーにターミナルでの手動実行を依頼）
