# team-info-daily-dev-memo スキル

## 役割
その日の `git` 履歴を確認し、team-info メンバー向けの共有文面を作る。

- 対象は当日のコミットと、その時点の `git status`
- ユーザーが日付を指定しない場合は、そのマシンのローカル日付を使う
- 出力は日本語で、短い段落ベースにまとめる

## 事前確認
1. `python3 .agent/skills/common/scripts/team_info_runtime.py worked-before-status` を実行する。
2. 結果が `known` なら続行する。`new` ならセットアップ導線を優先する。
3. `git status --short` で未コミット変更の有無を確認する。

## 収集する情報
日付を `YYYY-MM-DD` として、次を確認する。

```bash
git -C "$TEAM_INFO_ROOT" log --since='YYYY-MM-DD 00:00' --format='%ad %h %s' --date='format:%Y/%-m/%-d %H:%M'
git -C "$TEAM_INFO_ROOT" diff --stat "$(git -C "$TEAM_INFO_ROOT" rev-list -n 1 --before='YYYY-MM-DD 00:00' HEAD)..HEAD" -- . ':(exclude)node_modules'
git -C "$TEAM_INFO_ROOT" status --short
```

必要なら個別コミットも確認する。

```bash
git -C "$TEAM_INFO_ROOT" show --stat --summary --name-only <commit>
```

## まとめ方
- まず「今日は何を進めた日か」を 1 文で言う。
- 次に、変更を 3〜5 個のテーマへ束ねる。
- 各テーマの「何をしたか」には、触ったファイル名を 1〜3 個添える。
- 次に、「何が変わったの？」で、見え方・動き・使い方がどう良くなったかを書く。
- その変更で、今後どんな利点があるかも 1 文で入れる。
- コミットメッセージは補助として使い、実際の差分内容を優先する。
- 共有先がメンバー向けであれば、刺激の強い題材名・実在人物名・センシティブな表現は中立化する。
  - 例: `新しいショート動画案件`
  - 例: `バズ動画用の新規コンポジション`
- 露骨な語句や煽る語句は避ける。

## 出力フォーマット
ユーザー指定がない限り、次の形に合わせる。

```text
YYYY/M/D 作業報告
本日は、[全体テーマ] を進めました。

[何をしたか]
- `file-a.tsx`, `file-b.tsx` を直しました。
- [テーマ1の要約]
- [テーマ2の要約]

[何が変わったの？]
- 何がどう変わったかを、見え方や使い方まで含めて説明します。
- 今後どんな利点があるかも書きます。
- 必要ならテーマごとに1文ずつ追加します。

[Git 状態の一文]。
```

## Git 状態の書き分け
- `git status --short` が空なら:
  - `本日分の変更はGitに反映済みで、未コミット変更はありません。`
- 未コミット変更があるなら:
  - `本日分のコミットに加えて、ローカルに未コミット変更があります。`

## 品質ルール
- 文量は 5〜7 文程度を基本にする。
- メンバーが一読で把握できる粒度にする。
- 専門用語を詰め込みすぎない。
- 同じ話題を重複させない。
- 共有文面では、機能追加・素材整理・運用改善の順で並べると読みやすい。

## コミットがない日の扱い
当日にコミットがない場合は、その事実を先に明記する。

```text
YYYY/M/D 作業報告
本日は、Git に反映された変更はありませんでした。
[未コミット変更がある場合だけ、その旨を 1 文で補足]
```
