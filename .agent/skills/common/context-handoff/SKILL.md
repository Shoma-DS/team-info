# context-handoff スキル

## 役割
コンテキストが長くなったり、トークンが 95% に近づいたときに、
今のセッションで行った作業・知識・次のステップを引き継ぎ文書（MD）に書き出す。
次のセッションでは、その文書を読み込んで削除してから作業を再開する。

---

## 定数（このスキル全体で使う値）

```
引き継ぎファイルパス: /Users/deguchishouma/team-info/.context-handoff/handoff.md
引き継ぎフォルダ:    /Users/deguchishouma/team-info/.context-handoff/
```

---

## ステップ 0 — セッション開始時の確認（必須）

作業を始める前に、必ず引き継ぎファイルの存在を確認する。

```bash
ls /Users/deguchishouma/team-info/.context-handoff/handoff.md 2>/dev/null && echo "EXISTS" || echo "NOTFOUND"
```

### ファイルがあった場合（EXISTS）

1. ファイルを読み込む。
2. 内容を **すべてユーザーに告げる**（省略しない）。
   - 「前回の引き継ぎ内容をお伝えします」と前置きしてから読み上げる。
3. ユーザーに「このファイルを削除して続きから始めますか？」と確認する。
4. 承認を得たら引き継ぎファイルを削除する。
   ```bash
   rm /Users/deguchishouma/team-info/.context-handoff/handoff.md
   ```
5. 「次にやること」に書かれた内容から作業を再開する。

### ファイルがなかった場合（NOTFOUND）

通常通り作業を開始する。引き継ぎファイルは存在しない旨をユーザーに伝えなくてよい。

## ステップ 1 — コンテキスト圧縮（自動）

エージェントが「コンテキストが長くなってきた」と感じたタイミングで自動的に行う。
ユーザーへの確認は不要。

### 圧縮の判断基準

以下のいずれかに該当したら圧縮を検討する:
- 同じ話題が 3 回以上繰り返されている
- 解決済みのエラー・試行錯誤の記録が大量にある
- ファイルの内容をそのまま貼り付けたブロックが残っている

### 圧縮のやり方

圧縮は「削除」ではなく「まとめ」として行う。
- 解決済みの問題 → 「〜を解決した」という 1 文に置き換える
- 試行錯誤の詳細 → 「最終的に〜の方法を採用した」という結論だけ残す
- ファイル内容の引用 → 「〜ファイルを確認した。重要な点は〜」という要約に置き換える

圧縮後に「コンテキストを整理しました」とユーザーに一言伝える。

- ユーザーから「できるところまでやってから引き継いで」と指示された場合、限界までタスクを消化した直後にこのスキルを適用する

## コンテキスト圧縮の自動判断ルール

以下の条件を満たすときは、エージェントが自発的に圧縮を試みる。

| 状況 | 対応 |
|------|------|
| 解決済みエラーの詳細ログが残っている | 「〜を修正した」の 1 文にまとめる |
| 同じファイルを 3 回以上読み直している | 最新の読み取り結果だけ残す |
| 完了したタスクの詳細手順が残っている | 「〜を完了した」の結論だけ残す |
| ユーザーとの確認・承認のやりとりが積み重なっている | 最終決定だけ残す |

圧縮後は必ず「コンテキストを整理しました」とユーザーに伝える。

## よくある使い方

### パターン A: トークンが切れそうになったとき（自動）

エージェントが「残りトークンが少ない」と判断したとき、自動でこのスキルを実行し
引き継ぎファイルを作成して「次のセッションで続きができます」とユーザーに伝える。

### パターン B: 手動で引き継ぎを作りたいとき

ユーザーが「引き継ぎを作って」「handoff して」と言ったら即座にファイルを作成する。

### パターン C: 新しいセッションで続きを始めるとき

セッション開始時に自動で `.context-handoff/handoff.md` を確認し、
あれば内容を読み上げて削除後に続きを始める。

---

## 注意事項

- 引き継ぎファイルには秘密鍵・パスワードを書かない
- ファイルは 1 つだけ管理する（上書き保存）
- チーム開発モードでも `.context-handoff/handoff.md` を gitignore しておくことを推奨する
  （個人の作業状態なので、チームに公開しなくてよいため）
