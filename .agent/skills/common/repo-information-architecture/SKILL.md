# repo-information-architecture スキル

## 役割
- repo 全体を棚卸しし、情報設計の観点で「役割の重複」「命名の揺れ」「入口の分散」「死に筋の残骸」を見つける。
- いきなり移動せず、先に判断基準を明文化してから統合案を出す。
- 実作業では、索引ファイルとルール文書も含めて整合を取る。

## 最初にやること
1. `.dev-mode` を確認する。
2. `python3 .agent/skills/common/scripts/team_info_runtime.py worked-before-status` を実行し、`known` か確認する。
3. `git status --short` で既存差分を確認する。
4. `python3 .agent/skills/common/repo-information-architecture/scripts/repo_ia_inventory.py --format markdown` を実行し、圧縮された棚卸し結果を得る。
5. `RULES.md`、`AGENTS.md`、`.agent/skills/skill-finder/SKILL.md` のうち、今回の整理に関係する箇所だけ読む。

## 進め方

### 1. 現状の分類
- まず各パスを「正本」「入口」「入力」「出力」「実行環境」「案件固有」「個人用」「一時物」に分ける。
- 同じ役割を別名で持つディレクトリを一覧化する。
- `node_modules`、`.git`、仮想環境、キャッシュは探索対象から外す。

### 2. 統合候補の判定
- 詳細な判定基準は `references/ia-principles.md` を使う。
- 次の条件を 2 つ以上満たすものを統合候補にする。
  - 置いているものの種類が同じ
  - 参照される入口が同じ
  - 使う人や用途が同じ
  - 片方が旧名や暫定名になっている
  - 検索時にノイズを増やしている

### 3. 提案の作り方
- 提案は必ず「現状」「問題」「移行先」「移行理由」の 4 点で書く。
- 変更対象が広い場合は、一度に全部動かさず「命名統一」「入口統一」「不要物隔離」に分ける。
- AI の探索コスト削減を目的にする場合は、`rg` で引っかかる曖昧名や重複 basename を優先して消す。

### 4. 実変更
- 実際に移動・統合する前に、影響が大きい変更はユーザーへ確認する。
- 変更後は、少なくとも次を整合させる。
  - `RULES.md`
  - `AGENTS.md` の関連記述
  - `.agent/skills/skill-finder/SKILL.md`
  - その場所を参照する `SKILL.md` やスクリプト

### 5. 退行確認
- 整理が終わったら、repo 全体で損なわれた機能がないかを確認する。
- 少なくとも次を行う。
  - 変更したパスを `rg` で再検索し、古い参照が残っていないか確認する
  - 変更対象にひもづく `SKILL.md`、スクリプト、ランチャー、設定ファイルの参照先切れを確認する
  - 実行可能な検証コマンドがある領域は、その範囲のテストや dry-run を実行する
  - repo 全体にテスト入口がない場合でも、主要導線の import エラー、ファイル not found、壊れた相対参照が起きていないかを確認する
- 大規模整理では、確認結果を「確認した範囲」と「未確認の残り」に分けて報告する。

## このスキルで優先する判断軸
- 検索で見つけやすいこと
- 役割から保存先を推測しやすいこと
- AI が repo 全体を読む前に、候補を絞り込めること
- 旧構成との互換より、今後の保守コスト低減を優先すること

## 避けること
- 「とりあえず移動」で構造だけ変えて、索引やルールを放置すること
- 1 回の整理で unrelated な領域まで巻き込むこと
- 同名ディレクトリを残したまま「説明で運用」しようとすること
- 個人用・一時物・生成物・正本を同じ階層で混在させること

## 使う補助ファイル
- 判断基準: `references/ia-principles.md`
- 棚卸し: `scripts/repo_ia_inventory.py`

## 完了条件
- 重複した役割のフォルダ候補が説明可能な根拠つきで洗い出されている。
- 採用した構成案が命名、入口、索引の 3 点で一貫している。
- 変更した場合は、関連ルールと索引がすべて更新されている。
- 整理後に、変更影響のある範囲で機能退行チェックと参照切れチェックが完了している。
