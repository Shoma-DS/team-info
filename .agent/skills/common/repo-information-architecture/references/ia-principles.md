# Repo Information Architecture Principles

このファイルは、repo 整理時の判断基準を短く参照するためのもの。
`SKILL.md` に長文を持たせず、必要なときだけ読む。

## 1. 先に決めること
- そのパスは何の正本か
- 誰が使うか
- 入力か出力か
- 長期保存か一時利用か
- AI が探索するとき、候補を絞る助けになるか

## 2. 統合してよい条件
次のうち 2 個以上が当てはまるなら統合候補にする。

- 同じ種類の成果物を置いている
- 同じ入口やコマンドから参照される
- 読み手が「どちらを見るべきか」で迷う
- basename が重複して `rg` や `find` のノイズになっている
- 一方が旧名、暫定名、派生名である

## 3. 分けて残す条件
- 権限や公開範囲が違う
- ライフサイクルが違う
- ツールチェーンが違い、混ぜると誤操作が増える
- 物理的に大きい成果物と、軽い正本を分ける必要がある

## 4. 命名の基準
- 役割名を優先し、実装名や一時事情を名前に残しすぎない
- 旧構成があるなら、新名称を 1 つ決めて収束させる
- top-level に置くものは「見ただけで用途が分かる」ことを優先する
- 同じ語を別用途で使わない

## 5. AI 探索コストを下げる基準
- 同名ディレクトリを減らす
- 入口ファイルを少数に寄せる
- 正本と生成物を階層で分ける
- 「personal」「tmp」「old」「copy」などの一時語を散らさない
- 索引ファイルと実体を一致させる

## 6. 提案テンプレート
各候補は次の形で書く。

```text
現状:
問題:
移行先:
理由:
```

## 7. この repo で特に疑う場所
- top-level に同種の作業フォルダが複数ある
- `outputs/` に旧名と新名が共存している
- 同じ basename が別階層に再出現している
- `personal/` なのか共有正本なのかが名前だけで分からない
- スキルの索引と実体にずれがある

## 8. 整理後の確認基準
- `rg` で旧パス名を引き、参照が残っていないこと
- 変更したフォルダを参照するスクリプトが import / open / read に失敗しないこと
- 実行入口があるものは、最低 1 回はその入口で確認すること
- テストがない領域でも、ファイル存在チェックと主要コマンドの `--help` / dry-run / validate を使って壊れを拾うこと
- 結果報告では「確認済み」「未確認」「リスクあり」を分けること
