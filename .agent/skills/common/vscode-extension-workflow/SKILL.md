# vscode-extension-workflow スキル

## 役割
- VS Code 系拡張のソース repo がどこにあるかを特定する
- 使っているエディタごとの拡張配置先を特定する
- 既存拡張を更新するか、開発版を横に置くかを判断する
- 反映後にどこを再読み込みすればよいかを案内する

## 最初に確認すること
1. どのエディタで使うかを確認する
2. 既存拡張を改造するのか、新規拡張を作るのかを確認する
3. その拡張の `package.json` がどこにあるかを探す
4. インストール済み拡張の配置先を探す

## 対象エディタ
- VS Code
- Cursor
- antigravity
- VSCodium

## 基本方針
- まずソース repo を触る
- 反映先は「実際にユーザーが使っているエディタ」の拡張ディレクトリに合わせる
- 既存拡張がすでに入っている場合は、そのフォルダ名を優先して更新する
- 既存拡張がない場合だけ、`publisher.name-version` 形式の開発版フォルダや symlink を作る
- 反映後は `Developer: Reload Window` か再起動を案内する

## ソース repo の探し方
- 開いている workspace 配下を先に探す
- 見つからなければ、ホーム配下の開発フォルダ候補を探す
- `package.json` の `publisher` `name` `version` を読んで、対象拡張かを確認する

## 反映先の探し方
- 配置先パターンは `references/path-patterns.md` を使う
- 既存拡張フォルダがあるなら、そのフォルダを更新する
- 例:
  - `gera2ld.markmap-vscode-0.2.11-universal`
  - `publisher.extension-name-1.2.3`

## ローカル反映の順番
1. ソース repo でビルドする
2. 反映先の既存拡張フォルダを確認する
3. 次のどちらかで反映する
4. `dist/` や `package.json` を既存拡張へコピーする
5. またはソース repo への symlink を作る
6. エディタ側で再読み込みする

## どちらで反映するか

### 既存拡張を上書きする向き
- すでにその拡張を日常利用している
- 反映先が `*-universal` の既存フォルダで固定されている
- 実際に読み込まれる場所を外したくない

### symlink を置く向き
- まだその拡張が入っていない
- 開発版をすぐ差し替えたい
- ソース修正のたびに再コピーしたくない

## 反映後の確認
- 拡張一覧に同名拡張が二重で出ていないか
- 目的のコマンドや UI が増えているか
- `Developer: Reload Window` 後に変更が見えるか
- エラーが出る場合は DevTools Console と拡張 Host ログを見る

## 注意点
- `code` や `cursor` CLI がなくても、拡張フォルダへ直接反映できる
- エディタごとに拡張ディレクトリが違う
- `-universal` 付きフォルダが本体として使われていることがある
- ソース repo の場所と拡張の読み込み先は別物として扱う

## ユーザーに渡すコマンドの書き方
- 固定の `/Users/...` は避ける
- repo 内の案内は `TEAM_INFO_ROOT` を使う
- ホーム配下のエディタ拡張ディレクトリは `$HOME` を使う
