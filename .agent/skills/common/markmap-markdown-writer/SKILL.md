# markmap-markdown-writer スキル

## 役割
- 普通の説明文を、Markmap に向いた Markdown へ組み替える
- 長い文章を、階層の浅い見出しと短い箇条書きへ分解する
- 小学生でも追いやすい順番で説明を並べる

## 使う場面
- ツールの使い方説明を mindmap 化したい
- 設定手順を枝分かれで見せたい
- 役割、仕組み、手順、注意点を一枚の Markdown で整理したい
- URL や GitHub リポジトリを読んで、構造化した学習メモを作りたい

## 出力ルール
- 先頭に短い frontmatter を置いてよい
- 本文は `#` `##` `###` を中心に組む
- 1行1意味で書く
- 箇条書きは短くする
- 1つの箇条書きに情報を詰め込みすぎない
- テーブルは避ける
- 長いコードブロックは避ける
- Mermaid は使わない
- HTML は使わない
- 引用を多用しない
- 画像前提の説明にしない

## Markmap 向けの形

### 良い形
- 大見出しでテーマを分ける
- 中見出しで話題を分ける
- 箇条書きで具体例を書く
- ファイルパスは独立した1行にする
- URL も独立した1行にする

### 悪い形
- 1段落が長い
- 表だけで説明する
- 箇条書きが長文だらけ
- 1つの見出しに話題を詰め込みすぎる

## 推奨の章立て
- これは何か
- 何ができるか
- まず何を準備するか
- どう設定するか
- どう使うか
- どこを見れば分かるか
- 注意点
- まとめ

## 子ども向けにするときのコツ
- 難しい言葉を先に言い換える
- 1文を短くする
- たとえ話を少しだけ使う
- いきなり専門用語を並べない
- 手順は番号感覚で上から下へ流す

## ファイルパスの書き方
- リポジトリ内パスは独立行で書く
- 例
- `claw-empire/package.json`
- `claw-empire/.env.example`
- `claw-empire/src/App.tsx`

## 生成フロー
1. まず情報源を読む
2. 何の説明かを1行で決める
3. 大見出しを先に決める
4. 各見出しに短い箇条書きを入れる
5. 長い文を短く切る
6. Markmap で枝が深くなりすぎていないか見る
7. 最後に「おすすめの読み順」か「まとめ」を入れる

## テンプレート

```md
---
title: [タイトル]
format: markmap
---

# [テーマ]

## これは何か
- [ひとことで]
- [だれ向けか]

## 何ができるか
- [できること1]
- [できること2]

## まず準備するもの
- [準備1]
- [準備2]

## 設定する場所
### [場所1]
- [内容]
### [場所2]
- [内容]

## よく見るファイル
- `[path/to/file1]`
- `[path/to/file2]`

## 注意点
- [注意1]
- [注意2]

## まとめ
- [要点1]
- [要点2]
```
